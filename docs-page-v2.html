What's New in v2 NORT v2.0

NORT v2 moves beyond paper trading. The core signals engine, AI advice, and dashboard from v1 remain intact — v2 adds a real-money layer on top. Users get a Base smart wallet, can fund it via MPesa or card, bridge USDC to Polygon, and execute real Polymarket trades — all from the same Telegram bot and dashboard they already use.

v2 Overview

Everything in v1 still works unchanged. v2 adds these capabilities on top:

  • Base Smart Wallets — Every user gets a Privy Smart Wallet (ERC-4337) on Base. Funded via MPesa through Fonbnk or direct crypto. Gas is sponsored via Privy's Paymaster so users never manually buy ETH or MATIC.
  • USDC Bridge via LI.FI — USDC moves from Base to Polygon automatically before each real Polymarket trade. LI.FI selects the best route across Across, CCTP, and Stargate.
  • x402 Micro-Payments — Premium AI advice is gated behind a $0.10 USDC payment on Base. No subscriptions. One payment unlocks one premium advice call.
  • Multi-Agent AI (AI Agent v2) — Three specialised sub-agents (Sentiment, Risk, Technical) now collaborate before producing a final structured output. Conversation memory enables multi-turn follow-up questions.
  • Auto-Trade Engine — When confidence clears the configured threshold, NORT can execute a real Polymarket trade directly from the user's smart wallet — no custody, fully non-custodial.
  • Copy Trading — Leader users' high-confidence trades are automatically mirrored to opted-in followers.
  • Kiswahili Language Support — The bot can switch to Kiswahili for any command using /language sw.

Paper Mode Still Available

v2 does not remove paper trading. Users switch between paper and real mode via /mode paper or /mode real. Paper mode remains the default for new users.

The Money Flow

In v2, two separate money flows exist simultaneously. They never overlap:

x402 Payment Flow User's Base wallet → $0.10 USDC → NORT Treasury wallet (also on Base). Stays entirely on Base. Instant. Unlocks one premium advice query. The bridge is never involved.
Trade Execution Flow User's Base wallet → LI.FI bridge (~2 min) → USDC on Polygon → Polymarket CLOB signed order. NORT never holds the funds — money goes directly from user's wallet to Polymarket.

Non-Custodial

NORT never holds user funds at any point. The smart wallet belongs to the user. AutoTradeEngine constructs and signs a transaction from the user's wallet directly to Polymarket. NORT is the interface, not the custodian.

Architecture Changes

The three-layer architecture from v1 is unchanged. v2 adds two new external integrations and three new backend subsystems:

New: Base NetworkUser smart wallets live here. x402 payments settle here. Trade receipts are anchored here as on-chain proof.
New: LI.FI BridgeMoves USDC from Base to Polygon before each real trade. Called by the AutoTradeEngine — invisible to the user unless they opt into notifications.
New: Conversation TableStores multi-turn message history per user per market. Powers follow-up questions to AI Agent without losing context.
New: UserPermission TableStores each user's trade mode (paper/real), auto-trade toggle, max bet size, and copy trading settings.
New: AutoTradeEngineThe backend subsystem that checks confidence, checks permissions, prevents double-execution via idempotency keys, and calls the real trade route when all conditions pass.

Base Smart Wallets

Every NORT v2 user gets a personal Privy Smart Wallet (ERC-4337) deployed on Base — a self-custodied smart contract account. This wallet holds their USDC on Base, receives on-ramp deposits, and signs all trade transactions. NORT never controls it.

How Wallets Work

NORT uses Privy Smart Wallets — ERC-4337 account abstraction wallets deployed on Base. They offer four key advantages over standard EOA wallets for NORT's use case:

  • Gas sponsorship via Paymaster — NORT pays gas on behalf of users, so beginners never need ETH to transact.
  • Batched transactions — bridge + trade can be submitted as one atomic operation instead of two separate approvals.
  • Session keys — users can pre-authorise trading sessions so the bot can execute without prompting every time.
  • Social recovery — beginners can recover wallets via email or phone number, not seed phrases.

Privy smart wallets are counterfactual — they are created instantly off-chain and only deployed on-chain when the first transaction is sent. This means sign-up has zero gas cost. When a new user onboards, the wallet address is stored in their WalletConfig record and linked to their Telegram ID. All subsequent operations — bridging, x402 payments, real trades — originate from this wallet.

Key Management

Private keys are managed entirely by Privy — NORT's backend never stores or sees them. Signing happens via Privy's embedded wallet SDK. The defaultChain: base config in PrivyProvider ensures wallets are created on Base, not Ethereum mainnet.

MPesa On-Ramp

Users in Kenya and other MPesa markets can fund their Base smart wallet directly from their phone without a crypto exchange. The on-ramp flow is handled by Fonbnk:

  • User initiates a deposit from the NORT dashboard or Telegram bot
  • Fonbnk converts MPesa KES to USDC and deposits directly to the user's Base smart wallet address
  • NORT backend listens for the Fonbnk webhook, verifies the HMAC signature, then confirms the on-chain deposit and updates the displayed balance
  • Balances are displayed in the user's local currency (KES, NGN, GHS) — not raw USDC — using a server-side FX rate cache that refreshes every 60 seconds

Local Currency Display

All on-chain values are stored as USDC internally. The backend exposes a GET /fx/rates endpoint that converts amounts to the user's preferred currency before display. A balance of 50 USDC shows as KSh 6,475 for a Kenyan user — immediately meaningful, not raw crypto.

Gas Sponsorship

A common barrier for new crypto users is needing a gas token (ETH on Base, MATIC on Polygon) before they can do anything with their USDC. NORT v2 eliminates this entirely using Privy's Paymaster.

Privy's Paymaster is built into the smart wallet SDK. NORT funds a Paymaster contract with ETH on Base, and all user transactions on Base are routed through it automatically — the user only needs USDC, never ETH. Per-user daily spending caps can be set to prevent abuse.

Negligible cost

At current Base gas prices (~0.001 Gwei), sponsoring 1,000 user transactions costs approximately $0.10–$0.50 total. Even at 100,000 transactions/month, the gas budget is under $50 — negligible relative to the UX benefit of removing all gas friction.

USDC Bridge (LI.FI)

Polymarket runs on Polygon, not Base. Before any real trade can execute, the user's USDC must move from their Base wallet to Polygon. NORT v2 uses LI.FI — a bridge aggregator — to handle this automatically.

Why We Bridge

The user's wallet lives on Base. Polymarket's CLOB requires native USDC on Polygon — specifically the Circle-issued token at address 0x3c499c...3359. The two chains use different tokens even though both are called USDC. Sending the wrong token to Polymarket fails silently.

The canonical Base bridge (Base → Ethereum L1) has a 7-day fraud proof window — unusable for live trading. LI.FI's recommended routes (Across and CCTP) both deliver native USDC on Polygon in under 2 minutes with zero slippage.

Bridging strategy: bridge on trade, not on deposit

NORT uses lazy bridging — funds stay on Base until a real trade is placed. This avoids wasting gas if the user never trades, and keeps funds on Base where they are accessible. When a real trade is triggered, the AutoTradeEngine initiates a LI.FI bridge, shows a "Preparing trade…" progress indicator, waits for the bridge confirmation (~1–2 min via polling), then submits the Polymarket order.

Never use USDC.e

USDC.e is the old bridged version of USDC on Polygon. Polymarket requires native USDC only. LI.FI's routes are configured to deliver native USDC — never USDC.e.

LI.FI Route Selection

LI.FI is not a bridge — it is a bridge aggregator. When the AutoTradeEngine needs to move USDC, it calls the LI.FI API once. LI.FI queries all available bridges simultaneously, compares them by output amount, speed, and cost, and selects the best route automatically.

The backend calls LI.FI exclusively — the frontend never calls it directly. The flow is three steps:

Rate limit design

LI.FI's public API has rate limits (~60 req/min). Because bridge requests are user-specific and time-sensitive, the backend applies these mitigations:

  • Quote cache (30s TTL) — the same route quote is not refetched while the user is still on the trade confirmation screen
  • Backend queue — an asyncio.Queue with a semaphore of 3 concurrent LI.FI calls serialises simultaneous bridge requests
  • Exponential backoff on 429 — retry cadence: 1s → 2s → 4s → 8s
  • LI.FI partner key — apply before launch; rate limits are significantly higher on the partner plan (free to request)

Bridge Fallbacks

If the LI.FI API is unreachable, the AutoTradeEngine falls back to calling Across directly via its SDK. If Across is also unavailable, the trade is queued and retried — it is never silently dropped. The user receives a bot notification explaining the delay.

Bridge transactions are idempotency-protected at the backend level. If a transaction is submitted but the receipt confirmation times out, the backend checks the on-chain state before re-submitting — preventing double bridging of the same amount.

x402 Payments

x402 is an HTTP-native micro-payment standard. Instead of monthly subscriptions, users pay per premium query — $0.10 USDC on Base per call. The payment is verified on-chain before the premium advice is generated. No account management, no API keys.

What x402 Is

The name comes from HTTP status code 402 ("Payment Required"). When a user requests a premium resource, the server returns a 402 with payment instructions. The client pays on-chain and re-submits the request with proof of payment in the headers. The server verifies the proof and returns the resource.

In NORT, this maps to: the user requests /premium_advice, the bot tells them to send $0.10 USDC on Base to the NORT treasury address, the user sends the transaction, and then sends the transaction hash as proof. The backend verifies the hash on-chain and unlocks the premium advice for that query.

x402 stays on Base

x402 payments never cross the bridge. They settle entirely on Base — from the user's Base wallet to NORT's Base treasury. They are completely independent of the trade execution flow.

Flow A — Manual Proof (Telegram)

This is the primary flow for v2. It involves four steps:

  • Step 1: User sends /premium_advice <market-id>. Backend checks the Payment table — no confirmed payment found. Bot replies with the NORT treasury address and the required amount ($0.10 USDC on Base).
  • Step 2: User sends 0.10 USDC from their Base wallet to the treasury address. Their wallet returns a transaction hash.
  • Step 3: User sends /pay <tx_hash> to the bot. Backend calls POST /x402/verify with the hash and user ID.
  • Step 4: The verifier fetches the transaction receipt from Base, parses the USDC Transfer event, checks recipient address, checks amount, checks the hash has not been used before (replay protection), and stores the receipt in the Payment table. Premium advice is then generated and returned.
/premium_advice market-001
→ Bot: "Send $0.10 USDC on Base to 0xNORT_TREASURY. Then /pay <tx_hash>"

/pay 0xabc123def456...
→ Bot: "Payment verified ✓ — generating premium advice..."
→ [AI Agent full analysis returned]

Flow B — Automated Client (Stretch Goal)

For fully automated integrations (such as Coinbase AgentKit or other x402-compatible clients), the payment proof can be attached directly to the HTTP request as an X-Payment header. The server verifies the EIP-712 signed payment authorization inline — no /pay command required.

This flow is intended for programmatic access to NORT's premium advice endpoint. The client signs a payment authorization specifying the amount, recipient, and expiry timestamp. The server recovers the signer address, checks it matches the user's registered wallet, validates the amount and expiry, and verifies no replay.

Security & Edge Cases

Replay attackThe tx_hash field is UNIQUE in the Payment table. A second submission of the same hash is immediately rejected with "Proof already used."
UnderpaymentThe verifier checks amount >= REQUIRED_AMOUNT. A payment of $0.09 is rejected even if sent to the correct address.
Wrong recipientThe verifier parses the USDC Transfer event and checks the to address matches NORT_TREASURY_ADDRESS exactly. Payments to any other address are rejected.
Failed transactionThe verifier checks receipt.status == 1. A failed transaction still has a hash — it will be rejected.
Stale proofThe block timestamp is checked — proofs older than 10 minutes are rejected to prevent cached-proof attacks.
User shares proofThe proof is stored against the submitting user_id. Another user attempting the same hash is rejected because the hash is already consumed.

Multi-Agent AI

Multi-Agent AI orchestrates Technical, Sentiment, and Risk sub-agents in parallel for fast, secure, and personalized advice. Features include personalization & memory, robust architecture, auto-trading, security, and language support. See the sections below for details.

Architecture

  • Multi-Agent Orchestrator: The orchestrator runs three specialized sub-agents (Technical, Sentiment, Risk) in parallel using Python asyncio.gather for speed and determinism.
  • TechnicalAgent: Pure Python. Analyzes market signals (momentum, volume) and price action from Neon DB.
  • SentimentAgent: Calls Groq/Tavily for news and social sentiment, wrapping all external text in XML tags to prevent prompt injection. Produces a sentiment score (1–10) and flags viral narratives.
  • RiskAgent: Pure Python. Fetches live wallet balance and open positions from Intern 2’s endpoint, never from stale DB data. Assesses user risk, max position size, and exposure.
  • SynthesisAgent: LLM (Groq/Ollama/Claude) receives all sub-agent outputs and market data, then produces structured JSON advice (summary, trending reason, risk factors, suggested plan, confidence, and position sizing for premium calls). Never executes trades or fetches data directly.

Personalization & Memory

  • Sliding Window Memory: The AI Agent loads the last 5 messages per user and market from Neon DB, injecting them into the prompt for context. This enables multi-turn conversations and follow-up questions, while keeping the system stateless and scalable.
  • User Profile: Language preference, permissions, and trading limits are fetched fresh from Neon DB on every request. No in-memory user state is kept, ensuring scalability and up-to-date personalization.
How conversation sessions work:
  • /advice market-001 — starts a new conversation, stores a conversation_id
  • Any free-text follow-up (e.g. "What if BTC drops 10% first?") — bot detects the active conversation, loads history, sends full thread to the AI
  • /newadvice — clears context and starts a fresh session for the same or a different market

Context Window Cap

Conversations are capped at the last 10 exchanges to control token costs. Older messages are dropped from context but remain stored in Neon for audit history.

Language Support

  • Kiswahili & English: The entire bot and dashboard experience, including advice, confirmations, and errors, can be delivered in Kiswahili or English. Users switch language via /language sw or /language en. The preference is stored in Neon DB and affects all AI prompts and responses.
  • Accurate Data: All numerical and market data remains in English internally; only user-facing text is translated, ensuring accuracy regardless of language.

Auto-Trade Engine

AutoTradeEngine is the backend subsystem that converts AI advice into real or paper trade actions. It runs after every advice call — but only triggers an actual trade if the user has enabled auto-trading, the confidence threshold is met, and all safeguards pass.

How It Works

After the AI advice pipeline completes and produces a suggested_plan and confidence value, AutoTradeEngine runs the following check sequence:

  • Mode check: Is the user in paper mode or real mode?
  • Auto-trade enabled: Has the user turned on /enable_autotrade?
  • Confidence gate: Is confidence >= user's configured threshold (default 0.75)?
  • Idempotency check: Has this exact advice_id already triggered a trade? If yes, skip.
  • Bet size check: Is the computed trade amount within the user's configured maximum?

If all checks pass and the plan is BUY YES or BUY NO (not WAIT), AutoTradeEngine calls the appropriate route: POST /papertrade for paper mode or POST /trade/real for real mode.

Safeguards

Idempotency keyEach advice call generates a unique advice_id. AutoTradeEngine stores this in Neon before executing. If the same advice_id is seen again (e.g. from a retry), the trade is skipped. Prevents double-execution under network failures.
Confidence thresholdDefault is 0.75 (75%). Users can raise this to be more conservative. AutoTradeEngine never executes on a WAIT recommendation regardless of threshold.
Maximum bet sizeEach user sets a per-trade maximum in their UserPermission record. AutoTradeEngine caps the trade amount to this value even if the AI suggests a larger position.
WAIT hard blockIf the AI returns WAIT, AutoTradeEngine exits immediately. No trade is placed. WAIT is never overridden by any configuration.
Real mode confirmationFirst-time real mode trades require an explicit /confirm command. Subsequent real trades within the same session can auto-execute if the user has set auto_confirm=true.

Paper vs Real Mode

Users switch modes with /mode paper or /mode real. Paper mode is the default. The mode is stored in the UserPermission table and persists between sessions.

Paper Mode AutoTradeEngine calls POST /papertrade. Deducts from the $1,000 virtual balance in Neon. No real USDC moves. Full P&L and leaderboard tracking still apply.
Real Mode AutoTradeEngine calls POST /trade/real. The backend calls LI.FI to bridge USDC from Base to Polygon, then submits a signed order to Polymarket's CLOB API. Returns an on-chain transaction hash as proof. Real USDC is at stake.

Real mode uses real funds

When in real mode, AutoTradeEngine moves actual USDC from the user's Base wallet. This is not reversible. Ensure your confidence threshold and bet size limits are set appropriately before enabling real mode.

Copy Trading

Copy trading in NORT v2 lets users automatically mirror the trades of high-performing "leader" users. When a leader's AutoTradeEngine fires a real trade that clears the stricter copy threshold (0.80 confidence), the backend fans out the same trade direction to all opted-in followers.

Leader & Follower Model

Any user can become a leader if their portfolio performance qualifies them. Followers opt in to specific leaders — they choose who they copy and set their own per-copy bet size limit. Followers can opt out at any time.

LeaderA user whose advice sessions are marked is_leader: true. Their high-confidence AutoTradeEngine executions are eligible for fan-out to followers.
FollowerA user who has opted to copy a specific leader. They set their own copy_size_usdc limit. They receive a notification every time a copy trade fires.
Copy thresholdA copy trade fires only when the leader's advice has confidence >= 0.80 — stricter than the solo auto-trade threshold of 0.75. This is because bad leader advice gets multiplied across all followers.

Fan-Out Flow

When a leader's AutoTradeEngine fires and clears the copy threshold, the backend executes the following fan-out sequence:

  • Query the CopyRelationship table for all followers of this leader
  • For each follower, call POST /trade/real (or /papertrade if the follower is in paper mode) with the same market and direction but the follower's own copy_size_usdc
  • Each follower's trade is idempotency-keyed independently — a failure for one follower does not affect others
  • Each follower receives a Telegram notification: "Copied @leader's trade: YES on <market> at <odds>."

Followers set their own sizes

A follower copying a leader never bets the same dollar amount as the leader. They set their own copy_size_usdc — e.g. a follower might copy every leader trade at $10 regardless of how much the leader bets.

v2 API Reference

All v1 endpoints remain unchanged. v2 adds new routes for real trade execution, x402 verification, conversation management, user permissions, on-chain wallet balance, and copy trading. All new endpoints are available under the /api prefix.

Real Trade Routes

POST /api/trade/real Execute a real Polymarket trade. Requires market_id, outcome (YES/NO), amount_usdc, wallet_address, and advice_id (idempotency key). Triggers the LI.FI bridge and Polymarket CLOB submission. Returns a Polygon transaction hash.
GET /api/trade/real/{id} Get the status of a real trade by ID. Returns bridge status, Polymarket order status, on-chain receipt, and P&L if the market has resolved.
GET /api/wallet/onchain-balance Returns the user's live on-chain USDC balance on both Base and Polygon. Used by the Risk Agent and shown in the dashboard wallet panel.

x402 & Wallet Routes

POST /api/x402/verify Verify an x402 payment proof. Requires tx_hash and user_id. Fetches the transaction from Base, validates recipient, amount, and replay status. Stores a confirmed receipt in the Payment table if valid.
GET /api/x402/status Check whether the current user has a valid unexpired payment on file. Used by the bot to decide whether to show the payment gate or proceed directly to premium advice.

Conversation Routes

POST /api/conversation/message Send a follow-up message to an active advice conversation. Requires conversation_id, user_id, and message. Returns the AI's in-context response without re-running the full signals and search pipeline.
GET /api/conversation/{id} Retrieve the full message history for a conversation. Used by the dashboard chat panel to display the conversation thread.
DELETE /api/conversation/{id} Clear a conversation and start fresh. Equivalent to the /newadvice bot command.

Permissions & Copy Trading

GET /api/permissions Returns the current user's permission settings — trade mode, auto-trade toggle, confidence threshold, and max bet size.
PATCH /api/permissions Update permission settings. Accepts any subset of trade_mode, auto_trade_enabled, confidence_threshold, max_bet_usdc.
POST /api/copytrading/follow Follow a leader user. Requires leader_id and copy_size_usdc. Creates a CopyRelationship record. The follower will receive all future copy trade fan-outs from this leader.
DELETE /api/copytrading/follow/{leader_id} Unfollow a leader. Removes the CopyRelationship record. No further copy trades will be executed for this follower from this leader.
GET /api/copytrading/leaders Returns the list of available leaders with their current performance stats — win rate, net P&L, and streak. Used by the dashboard's copy trading discovery panel.

Telegram Bot

The NORT Telegram bot is the primary text-based interface for the system — every command, every payment, every response flows through here. It is implemented using python-telegram-bot (async) and runs in webhook mode in production. The bot contains no business logic — it formats requests, routes them to the correct FastAPI endpoint, and formats the response back to the user.

Architecture & Routing

The bot acts as a thin routing layer between the user and the backend. Every command maps to a specific FastAPI endpoint or service call. The bot itself never queries the database or calls the AI — those responsibilities belong to the backend and the OpenClaw agent respectively.

All requests take one of two paths through the system:

Fast Path Used for market data, signals, portfolio checks, and paper trades. The bot calls the FastAPI backend directly. No AI is invoked. Response is near-instant.
Agent Path Used only for /advice and /premium_advice. The request is routed through POST /agent/advice, which triggers OpenClaw. This path is slower and consumes AI tokens.

Webhook in Production

The bot is configured for webhook mode in production rather than long polling. The FastAPI app exposes the webhook endpoint, which is registered with Telegram. Async handlers ensure concurrent updates are processed correctly without blocking.

Onboarding & Wallet Setup

When a new user sends /start, the bot walks them through account creation and wallet authorization before any trading commands are available.

New user flow:
  1. User sends /start — the bot registers their Telegram ID and creates an account.
  2. The bot prompts wallet creation via the CDP SDK (handled by the wallet service).
  3. The bot prompts the user to run /authorize <max_usdc>, which sets an ERC-20 spending allowance on-chain. This allowance is stored in the database and is required before /premium_advice can deduct USDC.
  4. User is shown an overview of available commands and a link to the Mini App dashboard.

Wallet linking

Users can also connect an existing MetaMask wallet by running /wallet connect from the dashboard, which calls POST /wallet/connect. Once linked, the Telegram ID and wallet address are associated in the database, and the same paper balance and trade history are accessible from both interfaces.

No business logic in the bot

The bot only routes and formats. All scoring, trading, payment verification, and AI advice logic lives in the backend services. This guarantees that the bot and dashboard always show identical, consistent data.

Response Formatting

All bot responses are formatted for readability on mobile using Telegram Markdown. The formatting rules are enforced across all commands:

  • Market questions are displayed in bold. Odds and scores use monospace code formatting.
  • Trade confirmations include the market question, direction, entry price, and a Basescan link for any on-chain tx hash.
  • Wallet balances always show the USDC label. P&L is prefixed with + or -.

FAQs

Common questions about NORT v2's new features — real trading, x402 payments, copy trading, and conversation memory.

Real Trading

Is any real money involved?

In paper trading mode — no. All trades are simulated with a virtual balance. If you switch to REAL mode using /mode REAL, trades are executed on-chain via your Base wallet using the Polymarket CLOB API. The x402 payment for premium advice deducts real USDC on Base regardless of trading mode.

Does NORT hold my USDC at any point?

No. NORT is non-custodial. Your USDC stays in your Base smart wallet until the moment a trade fires — at which point it goes directly from your wallet to Polymarket via the bridge. NORT's backend constructs and coordinates the transaction but never controls the funds.

What happens if the bridge fails mid-transaction?

The AutoTradeEngine detects the failure, does not submit the Polymarket order, and queues a retry. It checks on-chain state before retrying to confirm the original transaction did not complete — preventing double-bridging. You receive a bot notification explaining the status.

How long does a real trade take end to end?

With Across as the bridge route (LI.FI's most common pick), bridging takes approximately 1–2 minutes. The Polymarket CLOB order submission takes a few seconds. Total end-to-end: roughly 2–3 minutes from AutoTradeEngine trigger to confirmed on-chain trade.

Do I need ETH to pay gas on Base?

No. NORT's Privy Paymaster sponsors gas for all Base transactions. You only need USDC — never ETH. Gas on Polygon for the Polymarket submission is either covered by the bridge relayer (Across bundles MATIC into the delivery) or by the Polymarket protocol directly.

Paper Trading

What is my starting balance?

Every new user starts with $1,000 in paper USDC. This balance is tracked against all your paper trades and is used to compute your P&L and leaderboard ranking.

How is P&L calculated?

P&L is recorded when a trade closes, based on the final market outcome versus your entry price. For open trades, mark-to-market value is calculated using current live odds. You can see this via /positions or the Trade page on the dashboard.

Can I exit a trade early?

Yes. Use /close <trade_id> to exit an open position at the current market price. The payout is calculated as shares × current odds, and the P&L is stored immediately.

Can I trade from both the dashboard and Telegram?

Yes. Trades placed via Telegram commands and via the dashboard are both recorded to the same account. Your portfolio, balance, and history are consistent across both interfaces.

AI Advice

Is basic AI advice free?

Yes. /advice <id> is available to all users at no cost and returns a brief summary with a recommendation. Full structured analysis via /premium_advice costs 0.10 USDC per query, deducted on Base.

Why does every advice response include a disclaimer?

It's a hard rule. Every OpenClaw response must include: "This is not financial advice. Paper trading only." This is enforced by the backend and cannot be omitted.

How accurate is the AI advice?

OpenClaw bases its analysis on current market data, signal scores, recent news, and social sentiment. It provides a structured perspective — not a guarantee. Always treat it as one input among many, not a definitive prediction.

Can OpenClaw see my trade history?

Yes. When generating advice, OpenClaw has access to your paper wallet summary via GET /wallet/summary. This allows it to factor in whether you already have an open position in the market being analysed, and to tailor position sizing guidance accordingly in premium mode.

What happens if my USDC balance is too low for premium advice?

The bot will reply with a friendly error message and prompt you to top up via /topup <amount_kes>. No USDC is deducted and no advice is returned until the balance check passes.
